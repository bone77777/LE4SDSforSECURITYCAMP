# セキュリティキャンプの課題の問一のレポです

## 背景

問題文に登場する AI エージェントは、深夜のインシデントにおいて以下のような判定を下しました。

- 通信量(4.7GB)と「サブネット単位での TI ヒット」のみを根拠に Confidence 87% と判定
- 一方で、デジタル署名が有効・親プロセスが `services.exe`(Windows 正規)・直近異常なし、という**安全側の証拠を判定にほとんど反映できていない**

この事案の本質は、**「悪性を示す弱い証拠」が「正規性を示す強い証拠」を上書きしてしまうスコアリングの非対称性**にあると考えました。本実装はこの非対称性を解消することを目的としています。

---

## 設計思想

問1-3 の本文で示した、3 つのコアを実装しました。

### コア 1: 証拠の解像度に基づく動的重み付け

脅威インテリジェンス(TI)のヒット内容を、その粒度によって重みを変えます。

- **IP 単位の完全一致(高解像度)** → `TI_IP_MATCH = 0.9`
- **サブネット単位の一致(低解像度)** → `TI_SUBNET_MATCH = 0.15`

サブネット一致は、あくまで「疑わしい近接環境」程度の背景情報として扱い、IP 単体の脅威評価とは明確に区別します。今回の事案では、ベンダー C のヒットがサブネット単位だったにもかかわらず Confidence High として等価に扱われたことが誤判定の原因となっていたため、ここを構造的に分離しました。

### コア 2: 安全を示す情報を強い負の重みとして定義

デジタル署名や正規の親プロセスといった「安全側の証拠」を、リスクスコアを劇的に引き下げる強い負の重みとして組み込みます。

- **有効なデジタル署名** → `TRUST_SIGNATURE = -0.85`
- **正規の親プロセス** → `TRUST_PARENT_PROC = -0.4`

異常検知系のモデルは陽性側に振れやすい構造を持っており、正常を示す証拠を確信度から「差し引く」項を持たないと、肯定証拠が一つヒットしただけで一気にスコアが跳ね上がってしまいます。本実装では、安全側の証拠が肯定的な証拠を相殺できる設計にしています。

### コア 3: 確信度に応じた段階的な対応(可逆性を考慮)

スコアに応じて、対応を 3 段階に分岐させます。

- `score >= 0.8` → ネットワーク遮断 + アカウント停止(高リスク)
- `0.3 <= score < 0.8` → アナリストへ通知(要確認)
- `score < 0.3` → 業務通信と判断し、ログを記録(安全)

問1-3 でも触れたとおり、「全遮断か、放置か」の二択ではなく**グラニュラーな対応**を取れる設計が必要です。本実装はその最小単位にあたります。

## 動作

### 入力(今回の事案を再現)

```python
incident_data = {
    "traffic_volume_gb": 4.7,
    "ti_match_type": "SUBNET",     # IPそのものではなくサブネット一致
    "is_signature_valid": True,    # デジタル署名は有効
    "is_authorized_parent": True   # 親プロセスは正規(services.exe)
}
```

### 出力

```
--- 判定プロセス開始 ---
[Behavior] 通信量(4.7GB): +0.56
[TI] サブネット一致(低解像度): +0.15
[Trust] 有効なデジタル署名を確認: -0.85
[Trust] 正規の親プロセスを確認: -0.4

最終確信度: 0.00%
【ACTION】 正常な業務通信と判断し、ログを記録します(安全)
```

問題のログでは Confidence 87% と判定されアカウント停止まで実行されましたが、本実装では同じ証拠から「正常な業務通信」と判定されました。**安全側の証拠を強い負の重みとして組み込めば、誤判定が回避できる**ことを示せたと考えています。

## 限界と今後の課題

このプロトタイプは線形和ベースの非常に簡易な実装で、以下の課題を抱えています。

1. **重みが固定値である**
   - 実運用では、ベイズ更新によって事後確率を継続的に更新する仕組みが必要です。
2. **正規の署名が盗用された場合に脆弱**
   - 攻撃者が正規署名を悪用する Living off the Land 攻撃には、本実装は逆に脆弱になります。問1-3 で触れた **Dempster-Shafer 理論やアンサンブル手法** で補強する必要があります。
3. **データソースが限定的**
   - 現状は通信量・TI・署名・親プロセスのみを入力としていますが、本来は EDR・NDR・IdP・CASB・ITSM(変更管理 DB)を統合した XDR アーキテクチャの中で動かすべきです。特に **ITSM からの変更管理情報**(SaaS 契約や月次決算スケジュール)を判定ロジックに注入できれば、今回のような業務サイクルとの不整合による誤検知は構造的に防げると考えています。
4. **シグモイド関数による正規化を簡略化している**
   - 現在は `max(0, min(1, raw_score))` でクランプしているだけです。本来はシグモイド関数で 0〜1 に正規化し、確率値として解釈できるようにすべきです。
